Why Port PAI to OpenCode?

Model Freedom

75+ providers. Local models via oMLX. Fully offline operation. No vendor lock-in.

Cost Control

Route cheap tasks to local Gemma. Expensive tasks to cloud Opus. Mix providers per agent.

Persistent Sessions

Daemon architecture survives disconnects, SSH drops, machine sleep. Resume where you left off.

The honest constraints: No guaranteed 1M context. Plugin API still maturing. No native multi-agent collaboration (yet). OpenCode's agent system is less sophisticated than Claude Code's вҖ” but it's evolving fast.

PAI's 7 Subsystems

Everything that needs to cross the bridge.

1. Algorithm

7-phase methodology. ISC criteria. PRDs. The gravitational center.

2. Skills

50+ self-activating domain experts. Progressive loading. SKILL.md standard.

3. Hooks

20 event-driven automations. Session capture. Voice. Tab titles. Rating capture.

4. Agents

9 named specialists with personas. Architect, Engineer, QA, Pentester, Artist...

5. Memory

Work tracking. Learning capture. Ratings. Research archives. State management.

6. Notifications

Voice TTS. Push (ntfy). Desktop. Duration-aware escalation.

7. Delegation

Multi-model routing. ComposeAgent traits. Parallel work. Spotcheck patterns.

Plugin Responsibilities

  • Write PRDs to memory/work/ on creation
  • Append ratings to signals/ratings.jsonl
  • Update state/current-work.json on transitions
  • Capture learnings on session end
  • Rotate and archive old entries

Key Difference from PAI

PAI uses Claude Code's native projects/ as the raw event firehose. OpenCode uses .opencode/data/ (SQLite). The plugin bridges this gap by intercepting events and writing to the PAI memory structure.

Key Architecture Decisions

ADR-1: Same Skill Standard

Port skills with minimal changes. Both platforms use agentskills.io. Skills become cross-compatible.

ADR-2: Algorithm = Rules + Skill

Behavioral rules in AGENTS.md (always loaded). 7-phase procedure as on-demand skill.

ADR-3: Memory as Plugin

No built-in equivalent. Custom plugin writes to PAI memory structure on events.

ADR-4: Local Models Default

oMLX for daily work. Cloud providers as quality overrides. Aligns with "scaffolding > model."

ADR-5: Notifications Unchanged

HTTP POST to localhost:8888 works identically. Zero adaptation required.

ADR-6: Native OpenCode Format

Rewrite agents in OpenCode frontmatter. Gains: mode, temperature, isolation, tools fields.

What PAI Gains from OpenCode

Run Anywhere

Not just Claude. Gemma, Qwen, Llama, GPT, Gemini. Local or cloud. Your models, your choice.

True Cost Control

70% of tasks on free local models. Research on cheap Gemini Flash. Architecture on cloud Opus. Per-agent routing.

IDE Integration

ACP connects PAI to Zed, JetBrains, Neovim. Use the same agents from terminal or editor.

Resilient Sessions

Daemon-based. Survives disconnects. Overnight batch processing. Fork conversations. Revert mistakes.

What PAI Loses (Honestly)

1M Context Window

Claude's 1M context is unmatched. Local models max at 128K. Requires more aggressive context management.

Dynamic Context Injection

<system-reminder> tags have no OpenCode equivalent. Context is less dynamic.

Claude's Code Quality

SWE-bench 80.8%. Local models don't match this. Cloud Claude still available but at cost.

Mature Plugin Ecosystem

Claude Code hooks are battle-tested. OpenCode plugins are still maturing. Edge cases exist.

The honest trade: You gain model freedom and cost control. You lose guaranteed quality and context depth. The architecture is designed so you can choose to use Claude via OpenCode when quality matters вҖ” you're not locked out, you're unlocked to choose.
